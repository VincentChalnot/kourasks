# KOURAKS — Interface Utilisateur (UI)

*Spécification des éléments d'interface et interactions.*

L'UI est construite avec **Pico.css** (esthétique minimaliste, presque terminal/système) et rendue réactive par **Alpine.js**.

## 1. Layout Principal

L'écran est divisé en grandes zones logiques :

### Header (Navbar)
À gauche :
- **Score :** Total des Kouraks possédés affiché en très gros.
- **Rendement actuel :** Affiché juste à côté en K/s (vert si positif, rouge si négatif).

Au centre :


À droite :
- **Vies (PDG) :** Confiance du PDG (ex: 2/3 annulations restantes).
- **Trimestre actuel :** Affiché en haut à droite.

*Interactions :*
- Cliquer sur le score ouvre une modale "Analytics" (graphiques d'évolution produits/dépensés si l'option est débloquée).

**Remarque importante:**
- Les marketplaces Hardware et Services sont bien distinctes, accessibles uniquement depuis le bouton "Déployer" situé à la fin de chaque liste respective (Hardware ou Services déployés).
- Aucun élément d'UI ne mélange hardware et service dans une même liste pour éviter toute ambiguïté, chaque interaction ou modale est spécifique au type d'élément (hardware ou service).
- La gestion de version n'existe que pour les services. Lorsqu’on navigue sur la marketplace Hardware, il n’y a aucune notion de version.

### Panneau gauche (Bloc supérieur) : Hardware déployé
Une liste verticale de cartes représentant les instances hardware déjà déployées.
- **Cartes Hardware :**
  - Affiche : Nom, Jauge de charge (Yield utilisé / `maximumYield`), Coût électrique.
  - Accordéon : En cliquant, on voit les services assignés à ce hardware avec leur statut et yield individuel.
- *Interactions :* Restart (si crashé), Shutdown, Vendre (modale).
- **Bouton "Déployer"** tout en bas de la liste : ouvre la marketplace Hardware pour ajouter un nouveau hardware.

### Panneau Gauche (Bloc inférieur) : Les Projets Actifs
Une liste verticale de blocs (cartes) représentant les projets.
- **Contenu du bloc projet :**
  - Titre (Gros) et Description (Petit).
  - En haut à droite : Kouraks totaux produits par ce projet | Rendement instantané | Objectif KPI en cours.
  - **Barre de progression KPI** : Visuel du trimestre en cours.
  - **Badges de Requirements** :
    - Vert : requirement satisfait et service `running`.
    - Orange : `deploying`, `restarting`.
    - Rouge : `crashed` ou `stopped`. Le projet complet est encadré en rouge.
    - Gris (bord rouge) : Aucun service assigné pour ce requirement.
- *Interactions :*
  - Clic sur un badge vide → Ouvre la Marketplace appropriée pré-filtrée sur ce type (Hardware ou Services selon le requirement).
  - Clic sur un badge actif → Ouvre une modale listant les services (et leurs versions) assignés de ce type.

### Panneau de droite : Les Services Déployés
- **Cartes Services :** 
  - Affiche : Nom, **Version**, Type, Statut (avec couleur/clignotement si crashé), Hardware cible.
  - Actions : "Start/Stop", "Restart" (si crashé), "Configurer" (Modale permettant de choisir la version).
- **Bouton "Déployer"** tout en bas de la liste : ouvre la marketplace Services pour ajouter/déployer un nouveau service.

### Footer (Terminal)
- Console scrollable fixée en bas (20% de l'écran).
- Affiche le flux continu des logs générés par le système (crashes, reboots, satires).

### Marketplace Hardware
- Liste dédiée à tous les hardwares disponibles au déploiement, triable et filtrable selon leurs attributs propres.
  - Nom : sort+filter
  - Type : sort+filter (multi-select)
  - Coût d'achat : sort+filter
  - Coût récurrent : sort+filter
  - Yield maximum : sort
- Actions : Déployer/Vendre, etc.

### Marketplace Services
- Liste dédiée à tous les services disponibles au déploiement, avec gestion explicite des versions.
  - Affichage groupé par famille de service : chaque service principal est repliable pour afficher ses différentes versions disponibles.
  - Pour chaque version : numéro/version, date de sortie, statut (stable/déprécié), notes.
  - **Filtre par défaut "Uniquement les dernières versions"** (pré-coché) : affiche seulement la version la plus récente de chaque service. Décochable pour voir les versions précédentes.
  - Filtres supplémentaires : Nom, Type, Gratuit/payant, Coût récurrent, Yield, Crash rate, Statut (déprécié ou non), requireHardware, global, etc.
- Les versions dépréciées sont affichées en rouge avec l’avertissement correspondant.
- Actions : bouton "Déployer" (ouvre la configuration avec la version présélectionnée).

### Modale de Configuration de Service
- Affiche les détails du service **et de la version sélectionnée** (description, yield, crash rate, date/version, notes de version, etc.).
- **Sélecteur de version** : possibilité de choisir, au moment du déploiement ou de la reconfiguration, la version à utiliser parmi toutes les versions encore déployables. (Par défaut, la dernière version stable est présélectionnée, ou la version actuellement déployée si la configuration concerne un service existant.)
- Si débloqué par un service : détail du calcul du yield effectif (multiplicateurs, surcharge hardware).
- Permet de changer le hardware cible (dropdown des hardwares disponibles) si applicable (requireHardware: true).
- Permet de changer le projet assigné (dropdown des projets disponibles) si applicable (global: false).
- Boutons annuler / déployer (applique les changements).

### Modale de Configuration de Service
- Affiche les détails du service (description, yield, crash rate...).
- Si débloqué par un service : détail du calcul du yield effectif (multiplicateurs, surcharge hardware).
- Permet de changer le hardware cible (dropdown des hardwares disponibles) si applicable (`requireHardware: true`).
- Permet de changer le projet assigné (dropdown des projets disponibles) si applicable (`global: false`).
- Boutons annuler / déployer (applique les changements).

**Note de cohérence :**
- La séparation stricte marketplace hardware/services, la gestion de versions côté services, les filtres spécifiques (dernière version, déprécié, etc.) et les modales adaptées garantissent l'absence d'incohérences ou contradictions dans l'UI.

## 2. Interactions Critiques

- **Crashes :** Quand un service crashe, l'écran doit le signaler visuellement (flash rouge léger sur la carte du service, badge projet passe au rouge, log dans le terminal). L'utilisateur doit manuellement cliquer sur un bouton "Restart" (tant qu'il n'a pas débloqué l'auto-restart).
- **Mails :** Pop-up modal qui s'affiche par-dessus toute l'interface. Le fond est grisé. Le jeu est en PAUSE totale (Alpine stop les ticks). Impossible de fermer sans cliquer sur une des options de réponse.
